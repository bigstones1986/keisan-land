# Substack編集長フロー

目的:

- Substack記事を毎回ゼロから考えない
- Search Consoleレポート、今日の作業記録、サイト改善内容から下書きの骨子を作る
- 公開は社長確認を残し、実際にない機能や言い過ぎを防ぐ
- 開発機能ではなく、初心者の困りごとと改善理由を主役にする

## 基本方針

- 完全自動公開はしない
- 自動化するのは「ネタ整理」「タイトル案」「本文下書き」「文末案内」「X投稿案」まで
- Chrome連携でSubstackへ下書きを入れる場合も、公開ボタンは社長が押す

## 毎回見る材料

1. `PROJECT_STATUS.md`
2. 最新の `search-console-private/reports/YYYY-MM-DD.md`
3. その日に公開または改善したページ
4. Search Consoleで見えた変化
5. 社長の言葉、違和感、悩み

## 記事の型

### タイトル

末尾に必ず付ける:

```text
（開発日記 #番号）
```

### 本文の流れ

1. 今日の入口
   - 社長が感じた困りごと、または今日の作業のきっかけ
2. 何を直したか
   - 専門用語を使わずに説明する
3. なぜ大事か
   - 保護者、先生、子どもにとっての意味を書く
4. まだできていないこと
   - 正直に書く
5. 次にやること
   - 小さく、具体的に書く
6. 文末案内
   - けいさんランド名とURLを必ず入れる

## 文末案内

毎回、最後に入れる:

```text
けいさんランドでは、小学生向けの無料計算ドリルを少しずつ作っています。
登録不要で、スマホでも印刷でも使えます。

けいさんランドはこちら:
https://keisan-land.netlify.app/
```

ステップ計算を紹介する回ではこちらも入れる:

```text
毎日ステップ計算はこちら:
https://keisan-land.netlify.app/step-calculation
```

## 編集長チェック

- 初心者がスマホで読んでも最後まで止まらないか
- 1文が長すぎないか
- 専門用語が残っていないか
- AIっぽいまとめ口調になっていないか
- 実際に存在しない機能を書いていないか
- 不安を煽っていないか
- 文末URLがクリックできる形になっているか
- `npm run qa:publishing` が合格しているか
- `AGENTS.md` の100点評価で90点以上か

## 現在の重点テーマ

タイトル案:

```text
文章題で止まったとき、最初に式を聞かない教材に変えました（開発日記 #12）
```

主題:

- 文章題を、場面、丸の図、式の順で考えられるようにした
- 毎回ちがう10問を、答え10までの範囲で作れるようにした
- 10,000問分の式、答え、出題範囲を確認した
- スマホで「しき」と「こたえ」が読みにくい箇所を直した

採用する切り口:

- 「式は？」の前に、場面を一緒に見る一段を置いた理由
