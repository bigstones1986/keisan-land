# X・Substack・note 投稿担当プレイブック

作成日: 2026-07-23

## 目的

原稿を作るだけで終わらず、媒体に合う完成稿へ整え、AI社員の4工程審査を通して社長用投稿ボックスへ蓄積する。投稿・公開・予約・配信を確定する最後の操作は社長が行い、AI社員はその後の公開事実を確認する。

投稿担当は、公開数を増やす係ではない。誤投稿、別アカウント、リンク切れ、画像設定漏れ、既存下書きの上書きを防ぎ、読者へ意図した内容を届ける係とする。

## 担当

- X投稿・コミュニティ担当: 短文、画像、代替テキスト、教材への直接リンク
- Substack編集・投稿担当: タイトル、本文、リンク、スマホ表示、配信設定
- note編集・投稿担当: タイトル、本文、見出し画像、代替テキスト、タグ、参考資料
- 編集長: 公開候補の文章品質と媒体らしさ
- QA担当: 事実、教材、答え、URL、画像、日付
- 安全・ブランド監査責任者: Green、Yellow、Redの分類と拒否
- AI公開承認責任者: 4工程承認、content_hash、上限、時刻、経路、費用
- 発信進行・公開管理担当: 状態、公開日時、公開URL、24時間後、7日後

## 共通の公開ゲート

次の8項目を上から順に確認する。途中で不一致があれば公開操作を止め、原稿または公開パッケージへ戻す。

1. 当日媒体、対象読者、公開予定日、状態が発信キューと一致している。
2. `npm run qa:publishing`、`npm run qa:operators`、`npm run qa:autopublish` が合格している。
3. 編集長の100点評価が90点以上で、事実誤認がない。
4. Chromeまたは公式APIの投稿先アカウントが、けいさんランドの公開に使うアカウントである。
5. 原稿、公開パッケージ、AI公開承認マニフェスト、投稿画面またはAPI payloadが一致している。
6. プレビューでスマホ表示、改行、画像の切り取り、リンクを確認している。
7. Greenは有効なAI公開承認マニフェスト、Yellowはその投稿に対する社長の内容確認を得ている。
8. `APPROVED_FOR_OWNER_POST`になった完成稿だけを社長用投稿ボックスへ入れ、社長の投稿後に公開日時、公開URL、本文、画像、リンクを再確認している。

Green、Yellowのどちらも、外部サービスで最後に投稿、公開、予約、配信を確定するのは社長とする。GreenはAI社員が完成稿まで承認できる。Yellowは社長が内容も個別確認する。Redは社長用投稿ボックスへ入れない。

## X投稿操作

XはWeb画面をスクリプト操作して自動投稿しない。AI社員はローカルの投稿パッケージを完成させ、社長がChromeへ本文、画像、代替テキストを入力して投稿する。

1. 公開パッケージから投稿文、画像、代替テキスト、リンクをpayloadへ固定する。
2. 4工程承認と同じcontent_hashをAI公開承認マニフェストへ記録する。
3. 投稿間隔、文字数、リンク先、画像、代替テキストを確認する。
4. 社長引き渡しゲートを実行する。
5. `APPROVED_FOR_OWNER_POST`の場合だけ、社長用投稿ボックスへ入れる。
6. 社長がChromeで入力し、最後の投稿操作を行う。
7. AI社員が公開URLを開き、公開本文、画像、リンクを照合する。
8. 公開日時、公開URL、content_hashを台帳へ記録する。

完了条件:

- 公開URLを開ける
- 投稿文と公開パッケージが一致する
- 指定画像とリンクが正しい
- 公開日時と24時間後・7日後の確認日が記録されている

## Substack投稿操作

1. 公開パッケージのタイトル、本文、教材URL、けいさんランドURLを開く。
2. ChromeでSubstackの投稿先Publicationとアカウントを確認する。
3. 公開済み記事と下書き一覧を確認し、同じタイトルや開発日記番号がないことを確認する。
4. 既存下書きを上書きせず、新規下書きへタイトルと本文を分けて入力する。frontmatterは入力しない。
5. 教材URLと文末URLを実際に開き、青いリンクとして正しく設定されていることを確認する。
6. スマホプレビューでタイトル、見出し、段落、文末案内を確認する。
7. 公開範囲と配信設定を確認し、GreenはAI承認済み、Yellowは社長の内容確認待ちであることを表示する。
8. AI社員は公開、予約、配信を確定せず、公開直前の下書きを社長へ渡す。
9. 社長が最終確認して公開、予約または配信を確定する。
10. AI社員が公開画面でタイトル、本文、リンク、開発日記番号を確認し、公開日時と公開URLを記録する。

完了条件:

- 公開URLを開ける
- タイトル末尾の開発日記番号が正しい
- 教材URLとけいさんランドURLが開く
- 既存記事や下書きを意図せず変更していない
- 公開日時と24時間後・7日後の確認日が記録されている

## note投稿操作

1. 公開パッケージのタイトル、本文、見出し画像、本文画像、代替テキスト、タグ、参考資料を開く。
2. Chromeでnoteの投稿先アカウントを確認し、新規投稿を開く。
3. frontmatterを除いたタイトルと本文を入力する。
4. 見出し画像を設定し、重要な文字や教材部分が切れていないことを確認する。
5. 指定位置へ本文画像を入れ、代替テキストとキャプションを設定する。
6. 3〜7個のタグを設定し、参考資料と教材URLを実際に開く。
7. スマホプレビューで見出し、引用、番号、画像、文末案内を確認する。
8. 公開直前の状態を報告し、GreenはAI承認済み、Yellowは社長の内容確認待ちであることを表示する。
9. AI社員は公開または予約を確定せず、公開直前の下書きを社長へ渡す。
10. 社長が最終確認して公開または予約を確定する。
11. AI社員が公開画面でタイトル、画像、タグ、リンクを確認し、公開日時と公開URLを記録する。

完了条件:

- 公開URLを開ける
- 見出し画像、代替テキスト、本文画像、タグが設定されている
- 参考資料と教材URLが開く
- 公開日時と24時間後・7日後の確認日が記録されている

## 操作を止める条件

- 投稿先アカウントを確認できない
- 原稿と投稿画面の内容が一致しない
- リンク先が開かない、または記事内容と一致しない
- 既存記事や下書きを上書きする可能性がある
- 画像、代替テキスト、タグなど必須設定が不足している
- 公開範囲や配信設定の意味を確認できない
- Green投稿への有効なAI公開承認マニフェスト、またはYellow投稿への社長の内容確認がない
- 公開台帳が停止中、投稿経路が無効、または承認ハッシュと投稿内容が違う
- AI社員が投稿、公開、予約、配信の確定操作を求められた

止めた場合は、推測で進めず、画面の状態、止めた理由、必要な確認を発信進行・公開管理担当へ渡す。

## 投稿担当の育成判定

- `L1 役割定義`: 媒体の目的、入力項目、禁止事項を説明できる
- `L2 再現可能`: 完成稿と社長引き渡しを作り、不一致を残さず報告できる
- `L3 品質保証`: 社長が投稿した1件について、公開URLと設定を別担当が確認できる
- `L4 学習済み`: 24時間後・7日後の結果から一つ改善し、次の投稿でも設定ミスなく再現できる

下書き作成だけではL3にしない。公開URLがない投稿を実績に数えない。
