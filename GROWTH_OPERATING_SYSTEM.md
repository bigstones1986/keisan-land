## 重点読者

第一読者:

- 小学1年生の子どもが、文章を読めても式にできず困っている保護者

第二読者:

- 10分程度で使える文章題プリントを探す先生、学童、支援者

読者の言葉:

- 「たすのか、ひくのか分からない」
- 「文章を読むと手が止まる」
- 「今日すぐ印刷できる10問がほしい」
- 「答えだけでなく、どう声をかけるか知りたい」

## けいさんランドの独自価値

- 登録不要、無料で、ページを開いてすぐ使える
- 答えだけでなく、短い考え方と保護者の見守り方がある
- 数字だけでなく、場面、図、式をつなげる
- 毎回ちがう問題を安全な学年範囲で作り、印刷できる
- 足し算だけ、引き算だけの練習後に、どちらの式を使うか考える混合10問へ進める
- 問題、式、答え、スマホ、印刷を公開前に確認する
- AIを補助に使っても、作成したまま公開せず項目別に確認する

## 媒体ごとの役割

### X

- 週5本を上限の目安にする
- 価値提供3本、会話1本、教材への直接案内1本
- 最初の2行に家庭の一場面を置く
- リンクは必要な投稿だけに入れ、内容が一致する教材へ直接つなぐ
- 投稿24時間後に表示と反応を記録する

### Substack

- 週1本
- 毎日20時30分に下書き作業を開始し、22時までに新規1本または既存1本の改善を本番下書きへ保存する
- `ready`が2本ある、題材が弱い、重複する、90点未満の場合は新規作成せず既存稿を改善する
- 運営者の実感、迷い、改善理由を人の言葉で書く
- 開発日記は週1回以内とし、主役を読者の困りごとへ移す
- 文末は記事で扱った教材へ直接案内する
- Substack内のRecommendationは、価値観が合う教育・子育て系だけを誠実に検討する

### note

- 週1本
- 毎日20時30分に下書き作業を開始し、22時までに新規1本または既存1本の改善を本番下書きへ保存する
- `ready`が2本ある、題材が弱い、重複する、90点未満の場合は新規作成せず既存稿を改善する
- 検索やnote内発見から長く読まれる保存版記事を作る
- 読者の疑問をそのままタイトルと見出しにする
- 冒頭で結論、2〜3文ごとの改行、具体例、図、出典を使う
- frontmatterへ媒体、公開予定日、状態を記録し、本文へ読者向け更新日を入れる
- frontmatterへ見出し画像、代替テキスト、タグも記録し、公開画面で設定漏れを防ぐ
- 見出し画像は記事内容と一致する実際の教材画面を優先し、横長の切り取りで重要部分が残るか確認する
- タグは3〜7個を目安に、学年、教科、単元、利用場面の具体語を組み合わせる
- 運営者が実際に見直した点を入れ、参考資料だけで終わらず今日試せる一歩で閉じる
- Substackの転載ではなく、検索意図に合わせて書き直す

### プロフィール導線

- 3媒体のプロフィールは、初めて知った人が「誰向けの何の発信か」を理解する入口にする
- X、Substack、noteの固定URLは `https://keisan-land.netlify.app/first-time.html` に揃える
- 教材を案内する個別投稿は、トップやプロフィールではなく内容が一致する教材へ直接つなぐ
- 表示名は運営者の人柄を残しつつ、「けいさんランド」を前半に含める
- 自己紹介は、対象読者、無料・登録不要・印刷OK、発信内容の順で短く書く
- 固定投稿はサイト全体の入口にし、古い社員数や実態とずれた機能を残さない
- 外部プロフィールの変更と固定投稿の差し替えは、実行直前に人間が確認する
- 月1回または重点テーマ変更時に、3媒体を同じ日に点検する

### けいさんランド本体

- 週1回まで、重点ページの主要要素を変更する
- 問題、図、解き方、見守り方、印刷、根拠を一つの教材体験にする
- 重点ページを置き換えず、次の学習段階が明確なときだけ関連教材を追加して内部リンクでつなぐ
- canonicalとサイトマップが `.html` のページは、NetlifyのPretty URLsを無効にして本番内部リンクも同じURLへそろえる
- Netlifyは `dist` だけを公開し、内部レポート、発信原稿、認証ファイル、品質検査を本番へ含めない
- 変更後7日間は、重大な問題がない限り同じtitleやh1を再変更しない

### 教育広報・自然な紹介

- `TRUST_AND_OUTREACH_PLAN_2026-07-16_TO_07-31.md` に沿って、読者が一致する紹介候補だけを調べる
- 相手の掲載方針を読み、勧誘や掲載依頼を受けない窓口には連絡しない
- 外部送信は週1件以下を目安にし、一斉送信せず、その相手向けに書き直す
- 法人格や事業者情報など申請条件を満たすと確認できない場合は送らない
- 自社のX、Substack、noteと第三者からの紹介を分けて記録する
- 成功は送信件数ではなく、実際の紹介、紹介流入、Search Consoleの外部リンクで判断する

### 利用者の声と導入支援

- 先生・支援者向けの感想窓口は `teacher-print.html` から `contact.html#teacher-feedback` へつなぐ
- 聞く内容は、学年・単元、利用場面、使いやすかった点、迷った点に絞る
- 子どもの氏名、学校名、成績、顔写真、連絡先を集めない
- 返信に必要なメールアドレスと、公開に使う許可を分けて扱う
- 感想や利用例は、内容と掲載場所を示して別途許可を得るまで公開しない
- 1件の感想を全体傾向にせず、事実、解釈、改善候補を分けて記録する
- 成功は感想数ではなく、実際の困りごとを教材改善へ反映し、確認後に使いやすくなったかで判断する

## AI社員間の引き継ぎ

### 検索露出

`分析担当 → 検索露出改善担当 → SEO担当・発信戦略責任者・教育広報・紹介担当 → QA担当 → 分析担当`

- 分析担当は、データ最終日と7日・28日の事実を渡す
- 検索露出改善担当は、詰まりと今週変える一つを決める
- 実行担当は必要な役割だけに絞り、同日に複数の主要要素を変えない
- QA担当は変更前後と変えなかった要素を確認する
- 7日後に分析担当が同条件で比較する

### 発信

`発信戦略責任者 → X投稿・コミュニティ担当 / Substack編集・投稿担当 / note編集・投稿担当 → 編集長 → QA担当 → 安全・ブランド監査責任者 → AI公開承認責任者 → 発信進行・公開管理担当 → 分析担当`

- 発信戦略責任者は、読者の困りごとと媒体を一つ決める
- X投稿・コミュニティ担当は、短い価値提供、画像、代替テキスト、直接リンクを投稿画面で照合する
- Substack編集・投稿担当は、運営者の実感が伝わる記事を新規下書きへ入力し、スマホ表示とリンクを照合する
- note編集・投稿担当は、検索から読める保存版記事を新規投稿へ入力し、画像、代替テキスト、タグ、リンクを照合する
- 編集長とQA担当は、読みやすさ、事実、教材、URL、日付を確認する
- 安全・ブランド監査責任者は、Green、Yellow、Redを判定し、拒否権を持つ
- AI公開承認責任者は、4工程の同一ハッシュ、投稿上限、予定時刻、鮮度、台帳を確認する
- Greenは承認済み完成稿として社長用投稿ボックスへ入れ、Yellowは社長の内容確認待ち、Redは拒否する
- X、Substack、noteの投稿・公開・予約・配信を確定する最後の操作は社長だけが行う
- 発信進行・公開管理担当は、公開日時、公開URL、24時間後、7日後を管理する
- 分析担当は結果を次のテーマへ戻す

### 教材品質

`教材研究・信頼担当 → 教材品質責任者 → QA担当 → 編集長 → 利用者リサーチ・導入支援担当`

- 教材研究・信頼担当は、学年範囲と根拠を渡す
- 教材品質責任者は、出題ルール、問題、答え、公開可否を判断する
- QA担当は、採点、スマホ、印刷、SEO、内部リンクを確認する
- 編集長は、サイト全体の方針と読者価値を確認する
- 利用結果や困りごとは、次の教材点検へ戻す

## 毎日の標準サイクル

### 朝

1. Search Consoleの直近7日と28日を確認する。
2. 重点ページ、重点検索語、表示回数、順位の変化を1行ずつ記録する。
3. 検索露出改善担当が、観測継続、検索改善、発信、紹介のどれを進めるか一つ決める。
4. 今日の読者の困りごとを1つ決める。
5. 発信日のみ、投稿先プロフィールの固定URLと自己紹介が現在の方針と一致するか確認する。

### 昼まで

1. X案、Substack案、note案、サイト改善のうち、その日に必要な1つを作る。
2. AI社員100点評価を行い、90点未満なら改善する。
3. 対象読者、今回直した1点、24時間後、7日後を原稿の公開管理情報へ記録する。
4. リンク先と書いている機能が一致するか確認する。

### 20時30分から22時

1. noteとSubstackの公開済み記事、下書き、投稿ボックスを確認し、重複を避ける。
2. 各媒体で新規下書き1本まで、または既存下書き1本の改善を進める。
3. noteとSubstackへ同文を転載せず、媒体別QAと100点評価90点以上を確認する。
4. Chromeが利用でき、正しいアカウントへログイン済みの場合だけ本番下書きへ保存する。
5. 公開、予約、配信、メール送信は確定せず、社長の最終操作を待つ。
6. 題材不足、重複、90点未満、`ready`2本以上の場合は新規作成を止め、改善結果と理由を記録する。

### 公開後

1. 24時間後にXまたは記事の反応を確認する。
2. 7日後に同じ曜日のSearch Consoleと比較する。
3. 成功、失敗、未判定を分け、次の変更を1つだけ決める。

### 下書き蓄積と公開進行

1. 予定日を過ぎた`ready`原稿を毎朝確認する。
2. 公開URLなどの証拠がなければ`verify`、未公開を確認できたら`reschedule`へ移す。
3. `published`は公開URLと日時を確認できた原稿だけに使う。
4. 24時間後と7日後は、実際の公開日から計算する。
5. 再設定する公開候補は一度に1本とし、過去分をまとめて公開しない。
6. AI社員は媒体ごとに1日1本を上限に下書き作業を行い、Xは最大5本、Substackとnoteは各2本を目安にする。上限到達後は新規稿ではなく既存稿を改善する。
7. AI承認を使う場合は、4工程の同一content_hashと社長引き渡しゲートの合格を確認する。
8. `ready`は「AI審査済み・社長の最終投稿待ち」とし、`OWNER_PUBLISHING_INBOX.md`へ優先順を記録する。
9. AI社員は投稿、公開、予約、配信を確定せず、社長の投稿後に公開URLと日時を記録する。

## 毎朝7:30の自動成長会議

- 自動運用名は「けいさんランド毎日成長会議」とし、同じ目的の自動運用を重複作成しない
- 日本時間の日付と日別カレンダーを確認し、当日の作業だけを対象にする
- Search Consoleはデータ最終日、7日、28日、重点ページ、重点検索語の順で読み、前回値と比較する
- クリック不足を、検索表示前、検索結果クリック前、教材開始前、外部発信からの流入に分ける
- その日に変える点は1つに絞り、大きな変更後7日間は同じ主要要素を再変更しない
- 当日のready原稿だけを、媒体別条件、自動QA、100点評価90点以上で確認する
- 予定日を過ぎたready原稿は、公開証拠を確認して`verify`または`reschedule`へ移し、当日原稿と混ぜない
- 90点以上は合計点だけでなく、具体的な読者、今回の編集焦点、確認日がそろった原稿だけに付ける
- 公開済み発信は24時間後と7日後の確認日だけを処理し、未公開原稿と混ぜない
- 作業後は当日の日次成長レポートと `PROJECT_STATUS.md` に事実、判断、結果、未確認事項、次回日を残す
- Greenの完成稿は社長用投稿ボックスまで進める。外部投稿の確定操作は行わない
- Yellowの公開、プロフィール変更、固定投稿変更、フォーム送信、サイトマップ送信、広告費使用は自動実行しない

## 毎日20:30の長文下書き運用

- 自動運用名は「note・Substack毎日下書き」
- 日本時間20時30分に開始し、22時までの完了を目標にする
- noteとSubstackは各1本まで新規作成または改善する
- 同文転載、重複、薄い題材、90点未満を拒否する
- Chromeでは本番下書き保存まで進め、公開、予約、配信、メール送信は確定しない
- Chromeが使えない場合はローカル原稿と公開パッケージの改善まで進め、外部保存済みと表現しない
- 完了後は下書きURL、評価、未確認事項、社長の公開待ちであることを記録する

## 週次判断

毎週水曜日に判断する。

- 表示回数が増え、順位も改善: titleを触らず、同じ読者テーマを発信する
- 表示回数が増え、順位は同じ: 内容と外部紹介を強化する
- 順位10位以内でCTRが低い: titleとdescriptionの検索表示を1要素だけ改善する
- 7日間表示がない: 内部リンク、インデックス、検索意図を確認する
- 数字が少なすぎる: 結論を出さず、次の7日間を観測する

## 禁止事項

- Green判定であっても、AI社員が外部サービスの投稿、公開、予約、配信を確定する
- XをWeb画面のスクリプト操作で自動投稿する
- 毎日すべての媒体へ投稿する
- 無関係な相手へ宣伝返信や相互リンク依頼を送る
- 競合記事を言い換えて量産する
- 子どもや保護者の不安を煽る
- Search Consoleの登録リクエストを連打する
- 順位や成績向上を保証する
